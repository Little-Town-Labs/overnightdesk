# Research: Use-Case Identity Foundation

## Existing State

The platform already has an appropriate opaque identifier: `instance.id` is a
UUID. It also has `instance.userId`, which models exactly one owner, and a text
`instance.tenantId` generated from the user's ID. That text value is reused for
subdomains, container names, callbacks, Phase paths, and Hermes routing.

The Go orchestrator separately creates a UUID `tenant_id`, plus a slug and
container name. The two UUIDs are generated by different systems and therefore
cannot be assumed to match. Hermes provisioning currently receives the text
`tenantId` and derives several infrastructure names from it.

This means the missing feature is not “add an ID.” It is an authoritative
relationship model and compatibility registry that makes existing IDs and
resource strings unambiguous.

## Decision 1: UUID is canonical; number is human-facing

Sequential numbers are useful in speech, support, URLs, and durable labels.
They are poor security identifiers because they are enumerable and invite
authorization by possession. The use case therefore keeps a UUID primary key
and may receive a unique immutable number. The number is never reused and is
not accepted as proof of access.

The owner confirmed that allocation is zero-based: OvernightDesk/Walter is
`Tenet 0`, Mitchel/Trevor is `Tenet 1`, and TTS/Titus is `Tenet 2`. These are
approved targets pending audited database allocation. Existing `tenant-0`,
`tenet-0`, and `tenet0-postgres` infrastructure labels remain compatibility
resources; Trevor data hosted by one of those resources does not make Mitchel
Tenet 0.

## Decision 2: Standardize `use_case`, reserve `Tenet N` for display

`tenant` already describes existing customer-tenancy fields. `tenet` sounds
the same but means a principle, making code review and incident communication
error-prone. The canonical schema/API term is `use_case`. The product may show
`Tenet N` as a deliberate human-facing label, but code must not create a second
ambiguous `tenet_id` beside legacy `tenant_id`.

## Decision 3: Model membership instead of ownership

The exact `instance.userId == session.user.id` rule is safe for a single owner
but cannot represent Titus shared by Gary and Austin. Membership is a separate
entity with role, state, scope, and lifecycle timestamps. A single-owner view
can remain as a compatibility projection while consumers migrate.

Titus currently operates for Gary without Teams. The planned canonical Gary
membership also does not depend on Teams. Austin's later membership and Teams
access are a separate authorization change, not a prerequisite for the
standalone Tenet 2 runtime.

## Decision 4: Runtime, persona, and person remain independent

The previously accepted runtime rule is preserved: separate primary memory
requires a separate Hermes runtime. One runtime may host multiple persona or
profile assignments. Multiple people may be members of one use case. Neither
persona nor person name belongs in a canonical runtime key.

## Decision 5: Infrastructure strings are bindings

Container names, volume names, subdomains, Phase paths, OIDC clients, intake
routes, and external registry UUIDs can change at different times. Treating
them as resource bindings provides a compatibility window and auditable
retirement without changing canonical identity.

## Decision 6: Expand before any cutover

The foundation adds tables and nullable references first. It backfills one
use case, compares resolution paths, and moves one authorization surface at a
time. No number or prefix is inserted into production resource names during
this feature. Infrastructure naming can be reconsidered later based on actual
operational value.

## Decision 7: Provision identity before a person accepts access

A use case, runtime, persona, and infrastructure bindings describe platform
state and do not require a person to have registered. Their audited foundation
transaction therefore creates zero memberships. A second transaction may add
an active membership only after the intended person has an email-verified
Better Auth subject. This keeps customer participation from blocking platform
delivery without using email as identity, inventing a user, or granting an
operator temporary access to somebody else's use case.

## Rejected Alternatives

- **Use the sequential number as the primary key**: enumerable, leaks creation
  order, and couples external references to allocation policy.
- **Rename every resource to contain the number**: large flag-day migration
  with no identity-security benefit.
- **Use persona as identity**: a runtime can host multiple personas and persona
  labels can change.
- **Use the Phase App as identity**: the two approved Apps intentionally group
  several resources and are the secret blast-radius boundary.
- **Reuse one existing UUID everywhere without a mapping**: the frontend and
  orchestrator already generate independent IDs.
- **Keep `instance.userId` as the permanent authorization model**: cannot
  express Titus collaboration or scoped roles.
